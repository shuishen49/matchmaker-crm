# Matchmaker CRM 交付计划

## 目标
把当前项目补齐成一个 **可本地完整预览、可作为 GitHub 仓库展示、最终可发布到 GitHub Pages** 的静态 CRM 演示站。

当前阶段优先级：**本地完整度 > README 展示效果 > GitHub Pages 发布**。

---

## 总原则
- 不接受“开发中 / 敬请期待 / 暂无数据 / 占位图”作为交付状态
- 所有主页面必须有真实可见内容，哪怕先做简单版
- 先让本地 `npm run dev` / `npm run preview` 都能稳定看，再处理线上
- README 要像项目介绍页，不像技术说明书
- 所有补的数据要统一口径，避免首页、表格、统计彼此打架

## 最低交付标准
### 1. 页面完整性
- 每个一级入口都能进
- 每个重点二级页面都能看到实际内容
- 不出现空白页
- 不出现“功能开发中”文案
- 不出现只有筛选表单但下方没数据的假完成页

### 2. 演示数据标准
#### 销售 / 待分配资源
- 8~12 条资源
- 字段至少包含：姓名、手机号、来源、标签、阶段、分配状态、最后跟进时间、负责人

#### 服务 / 待分配客户
- 6~10 条客户
- 字段至少包含：客户姓名、服务级别、阶段、约见状态、顾问、最近动作

#### 财务
- 合同管理：6 条以上合同
- 收款管理：6 条以上收款记录
- 退款管理：4 条以上退款记录
- 红娘业绩：至少 4 位成员，带目标、完成额、完成度

#### 记录
- 跟进小记：8 条以上
- 导入记录：4 条以上
- 导出记录：4 条以上

#### 管理
- 部门管理：树状部门结构可看
- 角色管理：至少 5 个角色
- 红娘管理：至少 6 位成员，含部门、角色、手机号、状态、入职时间

#### 设置
- 基础设置：已有内容保留并优化
- 话术库：至少 6 条话术，分组展示
- 跟进模板：至少 6 条模板
- 服务套餐：至少 4 个套餐
- 我的账号：至少 1 个完整账号资料卡

### 3. README 展示标准
- 首页一句话简介
- 功能亮点
- 5 张左右项目截图
- 本地预览方式
- 最终在线地址（上线后补）

### 4. 本地验收标准
- `npm run dev` 正常
- `npm run build` 正常
- `npm run preview` 正常
- 本地访问不白屏
- 主菜单和关键子页面都能展示

## 工作拆解（按执行顺序）

### Phase 1：盘点与数据模型统一
**目标：** 先定演示数据口径，避免后面每页各写各的。

任务：
- [ ] 列出各页面共用字段（人员、资源、合同、跟进、套餐、模板）
- [ ] 统一演示数据命名风格（姓名、手机号、时间、状态）
- [ ] 先补出一套中心 mock data，尽量可复用到多个页面

产出：
- 可复用的演示数据结构
- 页面字段映射表

### Phase 2：干掉明显空壳
**目标：** 先清最刺眼的“开发中 / 暂无数据 / 空白页”。

优先处理顺序：
1. 设置页：话术库 / 跟进模板 / 服务套餐 / 我的账号
2. 管理页：红娘管理
3. 记录页：跟进小记 / 导入记录 / 导出记录
4. 财务页：把 0 值占位替换为真实演示数据

验收：
- [ ] 以上页面不再出现开发中/暂无数据
- [ ] 都有真实可视内容

### Phase 3：补齐业务演示感
**目标：** 让销售、服务、财务不像 UI 壳子，而像能演示业务流的系统。

任务：
- [ ] 销售模块关键页补足表格数据、标签、阶段状态
- [ ] 服务模块关键页补足客户、约见、服务合同数据
- [ ] 财务模块补齐合同、收款、退款、业绩看板
- [ ] 首页统计数字与业务表格口径尽量一致

验收：
- [ ] 待分配资源可看
- [ ] 待分配客户可看
- [ ] 合同/收款/退款可看
- [ ] 首页不再像纯装饰面板

### Phase 4：README 与截图重做
**目标：** 让仓库首页像成品项目。

任务：
- [ ] README 改为“简介 + 亮点 + 截图 + 本地预览 + 部署”结构
- [ ] 重新生成截图，确保覆盖关键页面
- [ ] 截图优先选最能体现完整度的页面

建议截图：
- 登录页
- 数据首页
- 待分配资源
- 待分配客户 / 约见管理
- 财务页
- 设置页（话术库或服务套餐）

### Phase 5：本地验收
**目标：** 先在本地彻底走通。

任务：
- [ ] 本地开发模式验收
- [ ] 本地静态预览验收
- [ ] 菜单逐页点击检查
- [ ] 检查是否仍有“开发中”“暂无数据”“0值纯占位”问题

### Phase 6：上线与线上验收
**目标：** 本地确认后再推 GitHub。

任务：
- [ ] 提交代码
- [ ] 推送 GitHub
- [ ] GitHub Pages 发布
- [ ] 线上检查资源路径与页面展示

## 角色分工（执行口径）
### 产品经理
- 控制范围，不让任务散掉
- 确认“哪些必须做，哪些可以先简化”
- 盯最终展示效果

### 程序员
- 直接补数据、补页面、清空壳
- 保证本地运行、构建、预览可用

### UI
- 调整表格、卡片、标签、信息层级
- 保证截图出来不像半成品

### QA
- 逐页点检
- 专查空白页、占位文案、假数据过少、统计冲突
